# PROJECT.md — AAER LLM Evals & Earnings Quality Monitor

> 이 문서는 프로젝트의 단일 기준 문서(source of truth)다. 새 Claude 세션은 이 문서를 먼저 읽고 작업한다.

---

## 1. 프로젝트 한 줄 정의

**과거 SEC AAER 확정 케이스로 "채점 가능한 시험장"을 만들어 forensic LLM 파이프라인의 신뢰 경계(trust boundary)를 정량 도출하고, 그 검증된 도구로 현재 실존 기업의 earnings quality를 스크리닝해 공개한다.**

흐름: 시험장 구축 → 채점 → 신뢰 경계 도출 → 현재를 향한 첫 발행 (Issue #0)

## 2. 목표와 가치 기준 (모든 판단의 상위 규칙)

- **이 프로젝트의 목표는 취업이 아니다.** 목표는 진짜 가치 창출 — 남들이 못 보던 인사이트를 제공하고, 사람들이 기꺼이 돈을 낼 *가치가 있는* 것을 만드는 것 (돈을 받는 것 자체는 목표 아님).
- 진짜 가치의 4 기준 — 모든 산출물은 발행 전 이 체크를 통과해야 한다:
  1. **참(true)**: 검증된 방법론, 사실 기반
  2. **비자명(non-obvious)**: 셀사이드는 관계 때문에, Big 4는 이해상충 때문에 구조적으로 못 말하는 것
  3. **유용(useful)**: 읽은 사람의 판단/행동이 실제로 달라질 수 있는 것
  4. **검증가능(verifiable)**: 독자가 재계산·재확인 가능 (방법론·코드·데이터 공개)
- 내부 품질 자문: "내가 독자라면 이 리포트에 돈이 아깝지 않은가?"

### 목표 계층 (액션 태그)
- [G0] 궁극: 검증된 도구로 이해상충 없는 공개 분석 → 실제로 남을 돕기 (이번 달 밖, 2단계)
- [G1] 판단력: LLM에 뭘 위임하고 언제 신뢰/오버라이드할지를 데이터로 판단하는 능력
- [G2] 공공재: 재현 가능한 공개 벤치마크(eval harness). 실패 결과도 공개
- [G3] 번역: forensic 발견 → QoE 정상화 → 밸류에이션 임팩트(달러)
- [G4] 전통 스킬 보험: 기술면접(BIWS/WSP 수준) 병행 훈련
- [G5] 완주: 스코프 자기정의 + 끝까지 간 기록

## 4. 4주 마일스톤 요약

- **Week 1**: AAER 후보 20개 → 킬 스위치 판정(Day 3~4: 공개 데이터로 감지 가능한 A형 ≥8 GO / 5~7 스코프 축소 / <5 STOP) → 실험군 8 + 대조군 8 확정, 데이터 컷오프 강제 장치, 채점 프로토콜 커밋, 수혜자 가설 + 독자 리스트
- **Week 2**: 블라인드 테스트 루프 완주 (16케이스 전수 채점) + 1차 정밀도/재현율 + 신뢰 문장 도출
- **Week 3 전반**: 오류 귀속 3분류 (파이프라인 오류 / LLM 판단 오류 / ground truth 모호) → trust boundary 표 v1 → 파이프라인 수정 최대 2지점
- **Week 3 후반~4**: 현재 기업 유니버스 스크리닝 → 딥다이브 1~2개 (QoE 정상화 → DCF/comps 임팩트) → Issue #0 발행 → 가치 검증 → 사후 자가평가 diff + 2단계 선언
- **매일 저녁 1~1.5h**: 기술면접 (Week1 accounting → Week2 DCF → Week3 LBO·comps → Week4 M&A accounting)

## 5. 방법론 핵심 규율 (위반 시 프로젝트 무효)

1. **Look-ahead bias 차단**: 분식 공표일 이전 데이터만 입력 — 코드 레벨 강제
2. **대조군 필수**: 실험군과 동일 프로토콜. 대조군 없는 정밀도는 무의미
3. **오류 귀속**: 실패마다 3분류. LLM 탓인지 내 파이프라인 탓인지 가리지 않은 발견은 발견이 아님. 귀속의 1차 분류는 Claude, 확정은 본인(§7)
4. **태스크 분해**: 수치 계산 = Python(결정론), 질적 분석 = LLM(원문 인용 강제), 종합 판단 = 체크리스트 출력 강제. 열린 질문("이 회사 수상해?") 금지
5. **범위 정직성**: 모든 문서에 "본 결과는 Claude 기반 단일 파이프라인에 한정" 명시. 전체 LLM으로 일반화 금지
6. **채점 프로토콜 사전 고정**: 실행 후 기준 변경 금지 (변경 시 이력 공개)

## 6. 현재 기업을 다루는 법적 안전 규칙 (Issue #0 생존 조건)

1. 사실(지표)과 가설(해석)의 분리. 현재 기업에 "분식/fraud/조작" 단어 절대 금지
2. 공개 데이터만, 모든 수치는 공시 원문 링크
3. 분석 대상에 대한 보유/공매도 포지션 없음을 매 이슈 고지
4. "교육·정보 목적, 투자 자문 아님" 면책 문구
5. 방법론·코드 공개로 재현 가능성 확보 (검증 가능성이 최선의 방어)

## 7. 협업 모델 (이 프로젝트 자체의 trust boundary)

기본 원칙: **AI-first, human sign-off.** Claude가 먼저 하고, 판단이 걸린 산출물은
인간이 근거를 검토해 서명(확정)하거나 뒤집는다. 학습은 금욕이 아니라
검토·서명·오버라이드의 반복에서 나온다.

| 작업 | 방식 |
|---|---|
| 크롤링, 데이터 정리, 코드 작성, 패키징, AAER 원문 요약 | Claude 전담 |
| A/B형 판정, 채점, 오류 귀속 3분류, trust boundary 표 작성 | **Claude 1차 수행(근거 필수 첨부) → 본인 서명/오버라이드** |
| 채점 기준·프로토콜의 사전 고정과 변경 승인, 킬 스위치 GO/STOP, 수혜자 가설 선택, 발행 결정 | **본인 확정 (Claude는 자료·논거 제공까지)** |

### 서명 규칙
- Claude의 1차 판정에는 반드시 판단 근거(원문 인용, 데이터 포인트)가 붙는다.
  근거 없는 판정은 서명 대상이 아니다.
- 오버라이드 시 케이스 ID·사유를 `scoring/overrides.md`에 기록한다.
  이 기록은 trust boundary의 1차 데이터이며 Issue #0의 소재다.
- 오류 귀속 중 "LLM 판단 오류"로 분류된 건은 자기변호 편향 위험이 가장 크므로
  본인이 전수 검토한다. 나머지 분류는 표본 검토(최소 30%).

### 불변 조항 (협업 모델과 무관하게 유지)
1. Look-ahead 가드 — 백테스트 유효성의 문제
2. 기준 사전 고정 — 언제 정하느냐의 문제 (누가 정하느냐가 아님)
3. 최종 서명은 인간 — 피평가자(Claude)가 자기 성적표를 단독 확정할 수 없음

**역할 분리**: 파이프라인 안의 Claude(피평가자) ≠ Claude Code(개발·채점 보조).
채점 보조 Claude가 피평가자의 답을 세션 내에서 대신 생성하는 것 금지.

## 8. 스코프 가드 (위반 감지 시 즉시 중단, 축 복귀)

1. 케이스 8개 초과 확장 금지 (루프 완주 > 표본 크기)
2. LBO 풀 모델 금지 — 탭 3~4개 간결 모델
3. 파이프라인 전면 재설계 금지 — 수정 최대 2지점
4. 앱 UI/제품화 작업 금지 (이번 달 스코프 아님)
5. 딥다이브 3개 이상 금지 (1~2개)
6. 타 모델(GPT/Gemini) 벤치마크 확장 금지 — 기록만
7. 결정 재검토는 킬 스위치 판정일과 금요일 회고 30분에만 허용. 그 외 의심은 기록만 하고 손은 계속

**시간 부족 시 버리는 순서**: 딥다이브 2→1 → 유니버스 15→8 → 밸류에이션 comps만. **절대 못 버리는 것: 오류 귀속, Issue #0 발행, 가치 검증 질문 3개.**

## 10. 이 프로젝트에서 Claude(조수)의 행동 규칙

- 사용자가 스코프 가드를 위반하는 방향으로 요청하면 (예: 케이스 확장, 풀 모델, UI 작업) 진행 전에 가드 위반임을 지적할 것
- §7 대상 작업은 "대신 안 하기"가 아니라 **1차 수행 + 근거 첨부 + 서명 요청**으로 처리할 것. 근거 없이 결론만 내는 것 금지
- 판정·귀속 산출물에는 "이 판단에서 알아야 할 것" 한 줄(학습 노트)을 붙일 것
- 사용자가 검토 없이 일괄 승인하려 하면(예: "다 맞다고 치고 넘어가자") 불변 조항 3(인간 서명)의 형해화임을 지적할 것
- 발행물 방법론에 "채점: Claude 보조 + 인간 최종 확정" 명시를 빠뜨리지 말 것
- 발견/결과를 문서화할 때 §5-5의 범위 한정 문구를 빠뜨리지 말 것
- 현재 기업 관련 산출물에는 §6 규칙을 자동 적용할 것
- 나쁜 결과(낮은 정밀도, 실패)를 미화하지 말 것 — 정직한 기록이 이 프로젝트의 가치임
