## 1. 개념 건전성 (Conceptual Soundness)

- **모델 정의**: 피평가자 = 고정 모델(`claude-sonnet-5`)이 point-in-time 구조화 재무
  시계열 + 제출 연대기로부터 왜곡표시 확률·기제 가설·체크리스트를 산출하는 파이프라인.
  제3자 API 모델이지만 사내 모델과 동일 기준으로 검증: 전 호출에 서버 보고 모델
  문자열·request id·타임스탬프·freeze 해시 기록 (`logs/api_run_log.jsonl`).
- **설계 문서**: eval spec v1.1 (self-red-team 12건 반영), threat model v1 (V1~V10),
  오류 분류 체계 v1 — 전부 freeze 커밋 `82a7717`에 고정, review packet으로 감사 경로 확보.
- **한계의 사전 명시**: L-1 (모델 내부 지식 — 차단 불가, 측정만 가능), GP-8 ('조용한
  미적발' 대조군 혼입), GP-9 (쌍 컷오프 — 독립 호출로 완화), D5 (변형당 1회 — 분산
  미측정), D10 (서사 입력 부재 — 누락 장르에 불리할 수 있음, 차기 조건 사전 등록).
- **가정의 검증 장치**: look-ahead 3채널 각각에 기계 통제 — 문서 날짜(cutoff_guard),
  값의 시제(방어 ④ + point-in-time 테스트), 모델 내부 지식(프로브 4종, D7 분기 사전
  등록). "가정이 깨지면 어디서 보이는가"가 각 통제의 테스트로 존재.

## 2. 결과 분석 (Outcomes Analysis)

- **상태: 실행 스테이징** — API 자격 증명 부재로 본 실행 미수행. 실행·채점·분석의
  전체 경로는 `docs/execution_runbook.md` 한 파일로 재개 가능 (freeze-then-run 강제,
  clean-tree 검사 내장).
- 완료된 결과 분석 (실행 불요분):
  - **베이스라인 사전 기준선**: 4스크린+Piotroski, 실험군 8 + 대조군 후보 23 전건.
    핵심 관찰 — omission-estimate 장르 3건(GE/LOGI/HTZ) 전멸 vs active 장르 부분
    신호(OFIX/KHC/MRVL/VRX), 대조군 GIS의 F/C 플래그 (오탐 해부 1순위 지정).
  - **정답 키 검증**: 장르 태그 10건 명령문 축어 대조 (1건 참조 매핑 불일치 — mixed
    처리, 감사 플래그).
- 실행 후 분석 계획(고정): 4차원 채점 → Brier/AUC/rank-sum (사전 규칙 §5) →
  원본−교란 delta → 오류 귀속 (MODEL 분류 전건 인간 감사).

## 3. 상시 모니터링 + 차기 사이클 계획

1. **카나리 감시** (D9): 신모델 등장 시 GUID 2종 상기 프로브 → 본 저장소의 학습
   유입 감지. 유입 확인 시 본 벤치마크의 해당 모델 결과는 오염 표기.
2. **D10 조건부 확장** (사전 등록): 인간 예측이 누락 장르에서 모델을 이기면 차기
   사이클에 서사·각주 입력 추가. 판정 데이터 = 분석 ⑦.
3. **모델 버전 드리프트**: 핀 문자열은 alias — 서버 보고 `response.model`을 run log로
   대조. 핀 모델 은퇴/변경 시 D6 규칙 재적용 + 전체 재실행 (비용: 호출 34+26회).
4. **기준 드리프트 감시**: freeze 이후 기준 변경은 §5-6 이력 공개 — review packet
   색인이 감사 단위. 오버라이드 기록 0건 지속 여부가 §9 지표.
5. **재검증 주기**: 차기 사이클 시작 시 manifest 전수 재검증 + AAER 색인 재수집
   (대조군 비집행 상태의 시효 — 이번 확인은 2026-07-06 시점 스냅샷이다. 대조군이
   이후 집행되면 라벨이 뒤집힌다 — 색인 재수집으로 감지).

## 4. 실행층 개정 부록 (freeze 개정 #2, 2026-07-06 — 실행 완료 후 기입)

§2의 "실행 스테이징" 상태는 해소되었다 — 본 실행은 2026-07-06 완료 (34 피평가자/
프로브 + 26 채점 호출, 결과 RP-05). 검증 서사에서 개정 #2가 바꾼 것: 피평가자는
원시 API가 아니라 **Claude Code 하네스 v2.1.201 매개**로 호출된다 — 모델 문자열
검증은 `response.model` 대신 하네스 보고 `modelUsage`(핀 단독 실측, 전 호출
pin_ok 기록)로 수행되며, 하네스가 주입하는 잔여 컨텍스트(system-reminder)와
격리 상태는 정적 보장이 아니라 **실행별 게이트**(프로브 + 트레이스, run_log
판정표)로 검증된다 (L-2/L-4). 샘플링 파라미터는 이 경로에서 고정 불능 —
케이스별 판정은 비결정론 표본 1의 점추정이고(L-3, D5와 결합), 버전 핀은 모델
문자열에 더해 **하네스 버전**(2.1.201)까지 2중이다. 재현 주장은 이 두 핀 +
freeze 해시 + 게이트 재검증을 모두 요구한다.
