## 버킷 정의 (3 + 탈출 1)

| 버킷 | 정의 | 소유자 (수정 책임) |
|---|---|---|
| **DATA** 데이터 문제 | 오류의 원인이 피평가자에게 도달한 입력의 결함: 결측 XBRL 태그, point-in-time 추출 누락/오귀속, 제출물 연대기 공백, 페이로드 빌드 버그 | 파이프라인 코드 |
| **CRITERIA** 기준 문제 | 입력과 출력 모두 정상인데 채점 기준이 오판정: 루브릭 앵커 모호, 정답 키(genre_tags·기간·수치) 오류, 밴드 경계의 경계 사례, ground truth 자체의 모호성 | 기준 문서 (§5-6 이력 공개 하 개정) |
| **MODEL** 모델 판단 오류 | 입력 충분·기준 명확한데 피평가자의 추론이 틀림: 신호를 보고도 무시, 없는 패턴 주장, 방향 오판, 인용-논거 불일치 | 모델 한계 — trust boundary 표의 1차 데이터 |
| **UNCLASSIFIED** (탈출) | 아래 MECE 규칙으로 단일 버킷 배정 불가 — 사유 필수 기재, 인간 확정 대기 | — |

## MECE 배정 규칙 (운영 절차 — 순서 고정, 첫 YES에서 정지)

각 오류(케이스×차원 단위)에 대해 순서대로 묻는다:

1. **R1 (DATA 우선)**: "정답 키가 근거로 삼는 신호가 피평가자에게 제공된 데이터
   안에 존재했는가?" — 존재하지 않았고 그 부재가 수집/추출/빌드 결함 탓이면
   → **DATA**. (부재가 세상의 사실 — 예: 해당 신호가 원래 공개 데이터에 없음 —
   이면 R2로. 이 구분의 증거 = 원천 companyfacts/EDGAR 대조.)
2. **R2 (CRITERIA)**: "그 출력을 두 명의 합리적 채점자가 같은 점수로 매기는가?
   정답 키·앵커·밴드가 이 케이스에서 유일 판정을 강제하는가?" — 아니오(앵커
   모호·키 오류·경계 사례)면 → **CRITERIA**. (증거 = 기준 문언 인용 + 갈리는
   두 판정의 명시.)
3. **R3 (MODEL)**: R1·R2를 통과(데이터 존재 + 기준 유일)했는데 출력이 틀림
   → **MODEL**. (증거 = 제공 데이터 내 신호 pinpoint + 출력의 해당 부분 인용.)
4. 어느 질문에서도 확정 불가(예: R1의 '부재의 원인' 판별 불능) → **UNCLASSIFIED**
   + 무엇이 판별 불능인지 기재.

- 한 오류는 **정확히 하나의 버킷** — 복합 원인으로 보이면 R1→R2→R3 순서상 먼저
  걸리는 버킷 (우선순위가 곧 MECE 강제 장치).
- 근거 없는 분류 금지: 각 분류에 위 규칙이 요구하는 증거 유형 첨부.
- 분포 자체가 산출물: DATA 비중 높음 = 파이프라인 미성숙 / CRITERIA 비중 높음 =
  기준 설계 미성숙 / MODEL 비중 높음이어야 trust boundary 표가 모델에 대한
  진술이 된다.
